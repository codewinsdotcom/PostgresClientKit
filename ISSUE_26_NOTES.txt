PROBLEM TO BE SOLVED

PostgresClientKit uses Foundation types (such as DateFormatter, Calendar, and DateComponents) to
convert date/time values from strings into more useful types.  This is non-performant.  Issue 26
identifies better approaches.

BASELINE

Let's benchmark the current implementation.  All performance testing is running on "My Mac" using
the Release build configuration (Swift Compiler - Code Generation Optimization Level "-O").  See
ExistingPerformance.swift.

    67.080 us PostgresValue(_:).timestampWithTimeZone().date
   172.154 us PostgresValue(_:).timestamp().date(in:)
   158.966 us PostgresValue(_:).date().date(in:)
   170.791 us PostgresValue(_:).time().date(in: tz)
   292.829 us PostgresValue(_:).timeWithTimeZone().date

POTENTIAL IMPROVEMENT

A couple of approaches (focused on PostgresTimestamp) were identified in Issue 26.

    21.363 us PostgresValue(_:).fastDateSwiftParser(timeZone:)
    22.599 us PostgresValue(_:).fastDateVsscanf(timeZone:)

ANALYSIS

For the date/time types, what are the inputs, outputs, and internal state?

            Inputs              Outputs             State   Lazystate
            
    PTSTZ   S, Ints+TZ, D       PV, DC, D           D       PV, DC
    PTS     S, Ints   , D+TZ    PV, DC, D(in:)      DC      PV
    PD      S, Ints   , D+TZ    PV, DC, D(in:)      DC      PV
    PT      S, Ints   , D+TZ    PV, DC, D(in:)      DC      PV
    PTTZ    S, Ints+TZ, D       PV, DC, D           D       PV, DC

We are most concerned about conversion from String to Date, since this is what is required to
process result sets.

For PTSWTZ, this currrently entails calling DateFormatter.date(from:) either once (if the String
contains fractional seconds) or twice (if not).  This takes about 65 us, which is nearly all of
the 67 us observed above.

For PTS, this currently entails:
- calling DateFormatter.date(from:) once or twice (as above) for a              50 us
  known time zone (UTC)
- calling Calendar.dateComponents(in:from:) to convert this Date to             21 us
  DateComponents
- extracting the DateComponents fields and using them to contruct a            < 1 us
  new normalized DateComponents
- calling DateComponents.isValid(in:) to validate the normalized                30 us
  DateComponents
- setting the calendar and timeZone properties of the DateComponents            11 us
- calling Calendar.date(from:) to get the Date in that time zone                37 us

This sums to 150 us, which is nearly all of the observed 172 us.


----------
2020-01-23
----------

IDEA

Our idea is to:
    - replace DateFormatter.date(from:) with a hand-rolled parser that produces a DateComponents
    - validate that DateComponents by round-tripping it to a Date without its fields changing
    - for PTSWTZ, doing this round-trip using the TimeZone that we predict the consumer will use
    - keeping a cache of Calendars for different TimeZones
